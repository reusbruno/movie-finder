Phase 2 (done): all six items shipped, each with its own verify-then-commit cycle.

1. ~~Free-text mood search~~ — built (`src/lib/mood-search.ts`, `src/app/api/movies/mood-search`, `src/app/api/tv/mood-search`). Claude Haiku 4.5 via structured outputs turns a query into genres (constrained to the real TMDB genre list)/keywords/reference titles/sort/year-range, resolved against TMDB (`with_keywords` added to `discoverMovies`/`discoverTV`) and fed into the existing discover pipeline. Degrades to a visible-but-disabled input ("Mood search — coming soon") when `ANTHROPIC_API_KEY` is unset; verified via lint/typecheck/build plus a real dev-server run (disabled state screenshotted, availability/validation/error-mapping branches exercised with a fake key against the live Anthropic API). Verified against the real Anthropic API on 2026-07-10 with `ANTHROPIC_API_KEY` finally added: "slow melancholic sci-fi" → sensible genres/keywords, "Interpreted as: …" caption renders. That real run surfaced a genuine bug: TMDB's `with_genres` was always OR'd (`|`), so a mood with 2+ genres (e.g. "Science Fiction, Drama") let pure-Drama titles with no Sci-Fi at all (Parasite, City of God, Mildred Pierce) into the results. Fixed by adding `genreMatchMode: "any" | "all"` to `discoverMovies`/`discoverTV` (`src/lib/tmdb.ts`) — the browse page's genre checkboxes keep `"any"` (OR, the standard multi-select convention), mood search now passes `"all"` (AND, genres are a hard filter for a mood). Also capped mood search to at most 2 genres (`MAX_GENRES_FROM_QUERY`) so an AND of 3+ loosely-related genres can't intersect to zero results, and the "Interpreted as: …" caption now derives from the same capped/resolved genre list actually used in the filter, not the raw uncapped LLM output. Re-verified live: every result for "slow melancholic sci-fi" now genuinely has both Science Fiction and Drama; a single-genre query ("funny heist movie" → Comedy, Crime) behaves as an AND of two and returns exactly what you'd expect; TV mood search checked too. Client-side mutual-exclusion UI (mood search clearing quick-search/genre/sort state and vice versa) still not separately verified. Two more real-usage fixes on 2026-07-10: (1) while a mood/blend/discover/search request was in flight, `MovieGrid` briefly showed its permanent "No movies found." message (the results state was still `null`, and `items = results ?? []` collapsed "no response yet" with "response was empty") - fixed generically in `MediaExplorer` (`resultsPending`, `src/components/skeletons.tsx`'s `SkeletonGrid`) since all four non-popular modes shared the exact same bug, not just mood search. (2) The AND-genre fix above still under-delivered for "cozy feel-good comedy for a rainy day": genre resolved to Comedy+Family AND'd, and TMDB's keyword tagging for abstract tone words ("cozy", "wholesome", "feel-good") turned out to be extremely sparse - measured directly against the live API, individual mood keywords covered as few as 5-26 movies in TMDB's *entire* catalog, so Comedy+Family AND'd with any of 5 such keywords collapsed to ~5 results (vs. ~1,746 for the same two genres with no keyword filter at all). Fixed in two parts: `discoverWithMoodFallback` (`mood-search.ts`, replacing the earlier `discoverWithGenreFallback`) now cascades keywords-dropped-first, genre-narrowed-to-primary-only as a last resort, since keyword tagging is the far sparser signal; and the `genres` field's LLM prompt now explicitly warns that TMDB's "Family" means children's content, not colloquial "wholesome", so it should only be picked when a query actually signals kids/family viewing - dropping keywords alone fixed the *count* but not the *flavor* (Comedy+Family sorted by popularity still surfaced Toy Story/Minions/Shrek), since "Family" itself was pulling toward children's content independent of keywords. The "Interpreted as: …" caption (both genre and keyword terms) reflects whichever filter set actually ended up applied. Verified live over several real LLM calls: "cozy feel-good comedy" no longer resolves to Family, typically landing on Comedy+Drama with 20 well-fitting results (Devil Wears Prada, The Intouchables, Forrest Gump; TV: The Bear, Shameless); an explicit "family movie night with the kids" query still correctly resolves to Family+Comedy (no over-correction); "slow melancholic sci-fi" (already had enough results) is unaffected; a 2s-delayed mood request confirmed no "No movies found" flash and a skeleton grid instead. See Known follow-ups for a `sort_by=popularity.desc` ranking-tone issue this pass didn't try to fix.
2. ~~Vibe blending~~ — built (`src/lib/vibe-blend.ts`, `src/app/api/movies/vibe-blend`, `src/app/api/tv/vibe-blend`, `src/components/title-picker.tsx`). Union of both titles' genres/keywords is the discover candidate pool (one page, 20 results); each candidate is scored per-signal (genre/keyword/recommendation-list membership), doubled-to-quadrupled when it hits *both* seed titles rather than one, and TMDB's own `/recommendations` overlap is weighted highest. New "Blend" mode in `MediaExplorer`, mutually exclusive with search/mood/filters, with a "Blending: X + Y" caption. Verified via lint/typecheck/build plus real TMDB calls (not mocked) through both the raw API and a driven browser session: Inception + Arrival → Interstellar top result; Breaking Bad + Better Call Saul → Dexter/The Sopranos; same-id, invalid-id, and missing-param cases all return clean errors. See Known follow-ups for the one thing still worth tuning.
3. ~~"Why this match" explanations~~ — built (`src/lib/match-explanation.ts`, `src/lib/keywords.ts`, `src/lib/anthropic-client.ts`, `src/lib/explain-match.ts`, `src/app/api/explain-match`). Deterministic explanation (no LLM) on every blend/mood-search/detail-page-recommendation card, reusing blend's genre/keyword scoring weights so ranking and explanation always agree; capped at 2 signals, "both seed titles" beats "one" on a tie, hidden entirely when nothing matched. New shared, cached keyword lookup (`src/lib/keywords.ts`, same TTL'd-Map shape as `ratings.ts`, no disk persistence) feeds blend/mood-search/detail-page recommendations alike. Optional on-demand Haiku elaboration via a new "Explain more" button (only rendered when `ANTHROPIC_API_KEY` is set), same shared Anthropic client now used by mood search too (extracted to `anthropic-client.ts`). Verified with real TMDB data — Interstellar blending Inception+Arrival produces the exact wording signed off on ("Shares Science Fiction with both Inception and Arrival — also spacecraft (Arrival).") — and a driven browser session confirming: plain browse/popular cards show no badge at all; blend/detail-page cards show the explanation; "Explain more" makes a real (non-mocked) call to Anthropic and degrades to "Try again" with the error in a title tooltip on failure, without navigating away from the grid. See Known follow-ups for what's pending the real API key.
4. ~~Where-to-watch~~ — built (`src/lib/watch-providers.ts`, `src/components/watch-providers.tsx`). New `getWatchProviders(mediaType, id)` in `tmdb.ts` hits `/movie|tv/{id}/watch/providers`; wrapped in `watch-providers.ts`, same TTL'd-Map cache shape as `keywords.ts` (in-memory only, no disk persistence - TMDB isn't the rate-limited API here), scoped to a hardcoded `"US"` region (see Known follow-ups). Rendered on the movie/series detail pages only (grid cards skipped - too small to be useful), grouped into Stream/Rent/Buy with provider logos and a "More info on TMDB" attribution link; renders nothing (not an empty box) when TMDB has no providers for a title. Verified via lint/typecheck/build plus a real dev-server run driven with Playwright against the live TMDB API: Interstellar (movie) and Breaking Bad (tv) both show correct real provider logos/names with no console errors (screenshotted); an obscure/invalid title (id 1) correctly renders no "Where to watch" section at all.

5. ~~Watchlist~~ — built (`src/lib/watchlist.ts`, `src/lib/use-watchlist.ts`, `src/components/watchlist-view.tsx`, `src/app/watchlist`, `src/app/api/watchlist/enrich`). Add/remove lives on the card itself (a bookmark icon, top-right of every `MovieCard`) rather than the detail page — cards appear in every context (browse, search, blend, mood search, recommendations, the watchlist page itself), so this is the one placement that lets a title be saved without leaving whatever grid it's already in; the detail page has no separate button, to avoid two controls doing the same thing. `watchlist.ts` stores only `{id, mediaType, title, posterPath}` per entry (no full TMDB objects); the new `/api/watchlist/enrich` route re-fetches details+ratings fresh on every watchlist page load (dropping any id that now 404s) rather than trusting a stale snapshot. New "Watchlist" nav link in `AppHeader`; `MovieGrid`/`MovieCard` gained an optional per-item `mediaType` so one grid can mix movies and TV (every other caller is still single-media and unaffected). Reads localStorage via `useSyncExternalStore` (`use-watchlist.ts`) - the first attempt used the classic mounted-state-plus-`useEffect` pattern, which tripped `react-hooks/set-state-in-effect` (bundled with Next 16/React 19); `useSyncExternalStore` is the built-in primitive for this exact "external store, no hydration mismatch" case and sidesteps that lint rule entirely. Hit and fixed one real bug along the way: `getWatchlist()` originally re-parsed JSON into a fresh array on every call, so `useSyncExternalStore` saw a "changed" snapshot every render and infinite-looped ("Maximum update depth exceeded") - fixed by caching the parsed array and only re-parsing when the raw localStorage string actually changes. Verified via lint/typecheck/build plus a real dev-server run driven with Playwright: empty state renders a plain message (not an error); saving two titles from the popular-movies grid shows filled bookmark icons and both appear on `/watchlist` reusing the exact same card grid; a real page reload preserves both (confirms localStorage persistence, not just React state); removing one drops it immediately, removing the last one restores the empty state; zero console errors throughout.
6. ~~"Surprise me"~~ — built (`src/components/media-explorer.tsx`). A "Surprise me" button sits right after the filter row in `MediaExplorer`, tied specifically to the genre/sort/rating filter state (`buildDiscoverParams`) rather than to whatever mode (search/mood/blend) happens to be on screen - a stray search query doesn't change what "respecting the active filters" means. On click it re-fetches page 1 of the same `discoverEndpoint` the filtered browse grid already uses (no new API route - explicitly reuses the existing discover pipeline), picks one result at random client-side, and `router.push`es straight to its detail page - the surprise is the pick, not a results view. No filters set still goes through the same call, which naturally resolves to the same broad `sort_by=popularity.desc` pool the Popular grid shows, so there's no separate "no filters" branch. Zero-result filters show a calm inline note ("No titles match the current filters.") rather than the red error text a real fetch failure gets, and don't navigate. Verified via lint/typecheck/build plus a real dev-server run driven with Playwright against the live TMDB API: no-filter click navigates to a valid random movie; with Action genre + Min IMDb 8 set, captured the actual `/api/movies/discover` request/response and confirmed both that the request carried `genres=28&min_imdb=8` and that the navigated id was a member of that response's results (not just "some navigation happened"); a forced-empty response shows the inline message with no navigation and no error; zero console errors throughout.

All six Phase 2 roadmap items are now complete.

## Known follow-ups

- OMDb's free-tier quota reset (2026-07-09) and ratings now populate correctly end-to-end - confirmed via a fresh blend request (Inception + Arrival, real `imdbRating`/`rottenTomatoesScore` values returned) and a detail-page visit. `getRatings`'s optimistic-marking dedup (see `src/lib/ratings.ts`) is unchanged by any of today's fixes; three concurrent requests for the same uncached title resolved consistently with no errors or races, but the exact "one OMDb call, not three" claim is still not directly instrumented (an attempt to log real call counts hit a shell-redirection issue with the backgrounded dev server, not a code problem) - low priority now that the broader "does this even work against the real API" question is answered, but worth a proper instrumented check if OMDb quota exhaustion becomes a recurring problem again.
- Mood search verified against the real Anthropic API on 2026-07-10 (see build plan item 1) - interpretation quality is good and the genre-AND bug it surfaced is fixed. Mutual-exclusion UI verified the same day: submitting mood correctly clears an active quick-search value and genre selection (confirmed via input value + `aria-pressed`, not just visually); typing in quick-search or toggling a genre while mood is active correctly resets `moodQuery`/results (heading and grid switch away from mood, re-verified this wasn't just a visual mask). That pass surfaced one real gap: `clearMood()` reset `moodQuery`/results/interpretation but not the mood text *input* itself, so the box kept showing stale text after another mode took over even though mood was no longer active - fixed by adding `setMoodInput("")` to `clearMood()` (`src/components/media-explorer.tsx`), verified against all three paths that call it (typing elsewhere, toggling a genre, the explicit "Clear" link).
- Mood search's `sort_by=popularity.desc` default (used whenever a query doesn't imply "best"/"acclaimed") ranks purely by genre/keyword membership plus raw popularity, with no notion of tone - so a popular title matching the resolved filters can be tonally the opposite of the mood. Concrete example from 2026-07-10, after the Family-genre and keyword-fallback fixes above: "cozy feel-good comedy for a rainy day" (resolved to Comedy+Drama) correctly surfaced Devil Wears Prada/The Intouchables/Forrest Gump/Life Is Beautiful/La La Land, but also Parasite and The Wolf of Wall Street - both genuinely popular Comedy+Drama titles, both about as far from "cozy feel-good" as the genre pair allows. Not fixed here; revisit as part of general ranking-quality tuning. Same "reasoned starting point, not tuned against real usage" situation as vibe blending's score weights below.
- Vibe blending's score weights (genre 1/2, keyword 2/4, recommendation 2/6 - see `src/lib/vibe-blend.ts` and `src/lib/match-explanation.ts`, which now share these) are a reasoned starting point, not tuned against real usage. Revisit once there's a sense of whether results skew too genre-heavy, too obscure, or too `/recommendations`-dominated. The OMDb burst this used to cause on every blend (enriching the full ~19-candidate pool) is fixed - ratings enrichment is now capped to the top 10 candidates *after* scoring/ranking (`MAX_ENRICHED_BLEND_RESULTS` in both `vibe-blend` routes), and the movie detail page's own rating now goes through `ratings.ts`'s shared cache (`getMovieRatings`) instead of calling OMDb directly, so a title already looked up via a grid/mood-search/blend costs nothing extra there. `blendTitles` still fires one TMDB keyword call per candidate (~20 per blend, parallelized, best-effort) - that's TMDB, not OMDb, and TMDB isn't the rate-limited API here, so it's left as-is.
- "Explain more" LLM elaboration quality verified against the real Anthropic API on 2026-07-10: for The Mandalorian and Grogu (matched via mood search on "epic space opera with a found family crew"), the deterministic "Matches Adventure, space opera." expanded to "Follow a lone bounty hunter across the galaxy as he becomes an unlikely guardian to a mysterious child in this thrilling space opera adventure. Their journey through dangerous worlds and impossible odds will keep you on the edge of your seat." - a genuine improvement, grounded in the title's actual premise (not invented detail). One Playwright gotcha hit while verifying: the button lives in a zero-height `grid-rows-[0fr]` panel until real CSS `:hover` expands it, so `click({force: true})` without hovering first silently no-ops - needed an actual `.hover()` immediately before `.click()`.
- The "why this match" deterministic wording verified against a real mood-search interpretation on 2026-07-10: "epic space opera with a found family crew" → Science Fiction/Adventure/epic/space opera/adventure/found family, and The Mandalorian and Grogu's card read "Matches Adventure, space opera." - a coherent, accurate subset. Results overall were genuinely on-theme (Dune, the Star Wars catalog, Guardians of the Galaxy, 2001: A Space Odyssey).
- Where-to-watch (`src/lib/watch-providers.ts`) hardcodes the `"US"` region rather than detecting it, since no region concept (geo-IP, user setting, `Accept-Language`, anything) exists anywhere else in this app yet. Fine for now, but means every user sees US availability regardless of where they actually are - worth building real region detection/selection once there's a reason to (e.g. non-US usage shows up), rather than continuing to hardcode.
