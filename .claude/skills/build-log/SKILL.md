1. ~~Free-text mood search~~ — built (`src/lib/mood-search.ts`, `src/app/api/movies/mood-search`, `src/app/api/tv/mood-search`). Claude Haiku 4.5 via structured outputs turns a query into genres (constrained to the real TMDB genre list)/keywords/reference titles/sort/year-range, resolved against TMDB (`with_keywords` added to `discoverMovies`/`discoverTV`) and fed into the existing discover pipeline. Degrades to a visible-but-disabled input ("Mood search — coming soon") when `ANTHROPIC_API_KEY` is unset; verified via lint/typecheck/build plus a real dev-server run (disabled state screenshotted, availability/validation/error-mapping branches exercised with a fake key against the live Anthropic API). Verified against the real Anthropic API on 2026-07-10 with `ANTHROPIC_API_KEY` finally added: "slow melancholic sci-fi" → sensible genres/keywords, "Interpreted as: …" caption renders. That real run surfaced a genuine bug: TMDB's `with_genres` was always OR'd (`|`), so a mood with 2+ genres (e.g. "Science Fiction, Drama") let pure-Drama titles with no Sci-Fi at all (Parasite, City of God, Mildred Pierce) into the results. Fixed by adding `genreMatchMode: "any" | "all"` to `discoverMovies`/`discoverTV` (`src/lib/tmdb.ts`) — the browse page's genre checkboxes keep `"any"` (OR, the standard multi-select convention), mood search now passes `"all"` (AND, genres are a hard filter for a mood). Also capped mood search to at most 2 genres (`MAX_GENRES_FROM_QUERY`) so an AND of 3+ loosely-related genres can't intersect to zero results, and the "Interpreted as: …" caption now derives from the same capped/resolved genre list actually used in the filter, not the raw uncapped LLM output. Re-verified live: every result for "slow melancholic sci-fi" now genuinely has both Science Fiction and Drama; a single-genre query ("funny heist movie" → Comedy, Crime) behaves as an AND of two and returns exactly what you'd expect; TV mood search checked too. Client-side mutual-exclusion UI (mood search clearing quick-search/genre/sort state and vice versa) still not separately verified. Two more real-usage fixes on 2026-07-10: (1) while a mood/blend/discover/search request was in flight, `MovieGrid` briefly showed its permanent "No movies found." message (the results state was still `null`, and `items = results ?? []` collapsed "no response yet" with "response was empty") - fixed generically in `MediaExplorer` (`resultsPending`, `src/components/skeletons.tsx`'s `SkeletonGrid`) since all four non-popular modes shared the exact same bug, not just mood search. (2) The AND-genre fix above still under-delivered for "cozy feel-good comedy for a rainy day": genre resolved to Comedy+Family AND'd, and TMDB's keyword tagging for abstract tone words ("cozy", "wholesome", "feel-good") turned out to be extremely sparse - measured directly against the live API, individual mood keywords covered as few as 5-26 movies in TMDB's *entire* catalog, so Comedy+Family AND'd with any of 5 such keywords collapsed to ~5 results (vs. ~1,746 for the same two genres with no keyword filter at all). Fixed in two parts: `discoverWithMoodFallback` (`mood-search.ts`, replacing the earlier `discoverWithGenreFallback`) now cascades keywords-dropped-first, genre-narrowed-to-primary-only as a last resort, since keyword tagging is the far sparser signal; and the `genres` field's LLM prompt now explicitly warns that TMDB's "Family" means children's content, not colloquial "wholesome", so it should only be picked when a query actually signals kids/family viewing - dropping keywords alone fixed the *count* but not the *flavor* (Comedy+Family sorted by popularity still surfaced Toy Story/Minions/Shrek), since "Family" itself was pulling toward children's content independent of keywords. The "Interpreted as: …" caption (both genre and keyword terms) reflects whichever filter set actually ended up applied. Verified live over several real LLM calls: "cozy feel-good comedy" no longer resolves to Family, typically landing on Comedy+Drama with 20 well-fitting results (Devil Wears Prada, The Intouchables, Forrest Gump; TV: The Bear, Shameless); an explicit "family movie night with the kids" query still correctly resolves to Family+Comedy (no over-correction); "slow melancholic sci-fi" (already had enough results) is unaffected; a 2s-delayed mood request confirmed no "No movies found" flash and a skeleton grid instead. See Known follow-ups for a `sort_by=popularity.desc` ranking-tone issue this pass didn't try to fix. One more real-usage bug on 2026-07-10: "modern romance" returned only 3 results including Forrest Gump (1994) and Indiscreet (1958) - the opposite of "modern". Root cause: `interpretMoodQuery`'s system prompt never told the model what today's date is, so it had no anchor to compute what "modern" should mean and fell back to treating it as a vibe keyword ("contemporary", "urban") instead of a year-range constraint - confirmed by inspecting three live runs, all of which returned titles spanning 1934-2026 with zero temporal narrowing regardless of the keyword-sparsity fallback above. Fixed by adding `` `Today's date is ${today}.` `` (computed server-side per request, not hardcoded) to the system prompt, and rewriting the `yearRange` schema's `gte`/`lte` descriptions to explicitly map fuzzy temporal language against that anchor - "modern"/"recent"/"contemporary" to roughly the last 10-15 years, "classic"/decade mentions to the right historical bounds - not just literal years. Also surfaced `yearRange` in the "Interpreted as: …" caption for the first time (`formatYearRange` in `media-explorer.tsx`) - it was being resolved and applied in the discover call already but was invisible in the UI and omitted from the API response's `interpretation`, which is exactly what made this bug hard to spot without querying the API directly. Verified live: "modern romance" now resolves `yearRange: {gte: 2011}` consistently and every result is from the last ~15 years; "recent thriller" → `gte: 2016`; "classic film noir" → `{gte: 1940, lte: 1958}` with results genuinely from that era (Touch of Evil, The Killing); "80s action movies" → exact `{gte: 1980, lte: 1989}` (Terminator, Scarface, Top Gun); caption confirmed rendering "1940–1959" etc. in a live browser session. One more threshold tuning pass the same day: "slow melancholic sci-fi" was still landing on exactly 3-5 results even after the keyword-drop fallback existed - measured directly against the live API, its 5 mood keywords OR'd, AND'd with Science Fiction+Drama, resolve to *exactly* 5 results, which satisfied `discoverWithMoodFallback`'s `MIN_MOOD_RESULTS = 5` threshold and never triggered the keyword-drop relaxation, even though 5 is clearly too thin - the same boundary trap as the "cozy" case's earlier borderline runs. Confirmed the fallback's relaxation direction itself was already correct (dropping keywords for this genre pair yields 291-1,687 results depending on vote-count threshold) - only the "is this good enough to stop relaxing" bar was too low. Also worth noting for future tuning: a "drop to fewer keywords" middle step (as opposed to all keywords or none) wouldn't help here even in principle - `with_keywords` is OR'd, so the union of any subset of keywords is always ⊆ the union of all of them; there's no keyword-count step between "all 5" and "none" that could ever yield *more* than the full-5 count already did. Fixed by raising `MIN_MOOD_RESULTS` from 5 to 10. Verified live over 4 repeated runs: "slow melancholic sci-fi" now consistently drops keywords and returns 20 results, genre still AND'd on Science Fiction+Drama, every title still genuinely fitting the tone even without the keyword requirement (Interstellar, Blade Runner 2049, The Prestige, Arrival, Stalker, Metropolis) - the genre pair itself already captures most of the "slow, contemplative" signal. "Cozy feel-good comedy for a rainy day" re-checked for regression: still 20 results, still no Family genre, across 3 runs.
